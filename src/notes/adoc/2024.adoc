== Open-Closed principle

Let's state the principle: "software entities (classes, modules, functions, etc.) should be open for extension, but closed for modification"
(see https://en.wikipedia.org/wiki/Open%E2%80%93closed_principle).

Here are my personal thoughts on how to achieve that. Let's consider a "set-theoretical" approach.
If we view a class as a set `A` of lines of code. Then we must not modify the content of `A`. This represents the closure aspect.

However, to extend the functionality of A, we will need to modify certain parts of A.
Therefore, we must (1) extract those parts from A as external sets, such as `B`, `C`, and so on.
Next, (2) establish a relationships `R1`, `R2` ... between the parts of `A` where code was extracted and the new sets `B` and `C`.

At this point, we have two options:

1. Extending the functionality by modifying `B`, `C`, etc., would contradict the Open-Closed Principle for `B`, `C`, etc.
2. Extending the functionality by introducing new sets (classes) such as `B'`, `C'`, etc., and defining relations `R1`, `R2`, etc.,
in a manner that allows seamless switching between `B` and `B'`, `C` and `C'`, etc.

image::set_A_A_R_B.png[]

I think option 2 aligns closely with the concept of "decoupling".

Mathematically, to enable the flexibility to switch between `B` and `B'`, `C` and `C'`, etc., the relations `R1`, `R2`, etc.,
must establish links between set `A` and families of sets `{B, B', B'', ...}`, `{C, C', C'', ...}`, and so forth. Like set-valued mappings.

For example the relation "use an object b of extendable class B" is a solution. Because if we have inside `A` the line

```
b.method1();
```

where `b` is of type `B`, then we can always replace `b` with an object of type `B'`, where `B'` extends `B`.

A more explicit relationship might be described as "use an object b of interface B" or "use an object b of abstract class B."
In light of the above, an interface serves as a relationship to a family or hierarchy of interfaces and classes that implement it.

A specific type of relation is the mapping, or function. If we put inside `A` the line

```
method1();
```

To extend the functionality of `method1()`, it must relate not only to its own body but also to a family of methods named `method1()``.
This can be achieved using the template method pattern.

Possibly, the last part involving the mapping concept could be refined to "use a method that can be overridden".

== How to avoid the N+1 Query Problem in GraphQL

- In https://piotrminkowski.com/2020/07/31/an-advanced-guide-to-graphql-with-spring-boot/[An Advanced Guide to GraphQL with Spring Boot], the author utilizes
`DataFetchingEnvironment.getSelectionSet()` to verify the presence of a specific field in the query and fetch (`Root.fetch` from Criteria API) the corresponding related entity:
+
[source,java]
----
if (s.contains("employees") && !s.contains("organization"))
         return repository.findAll(fetchEmployees());
----
+
[source,java]
----
private Specification<Department> fetchEmployees() {
   return (Specification<Department>) (root, query, builder) -> {
      Fetch<Department, Employee> f = root.fetch("employees", JoinType.LEFT);
      Join<Department, Employee> join = (Join<Department, Employee>) f;
      return join.getOn();
   };
}
----

- In https://github.com/dguduguntla/graphql-spring-jpa-spqr[GraphQL in Java using SPQR], the author uses a class `EntityGraphSelectorUtil` that analyze the fields
from `DataFetchingEnvironment.getSelectionSet()` and chose the right `EntityGraph` for subsequent fetching. See for eaxample
https://github.com/dguduguntla/graphql-spring-jpa-spqr/blob/master/src/main/java/com/gdp/graphql/service/EmployeeService.java[EmployeeService] class.
The project uses https://github.com/Cosium/spring-data-jpa-entity-graph[Spring Data JPA EntityGraph] as depenedcy to send a EntityGraph object as parameter to
Spring Boot repository methods.

- In https://github.com/dguduguntla/graphql-spring-jpa-spqr[GraphQL in Java using SPQR], the author employs a class named `EntityGraphSelectorUtil` to analyze fields from
`DataFetchingEnvironment.getSelectionSet()` and select the appropriate `EntityGraph` for subsequent fetching. For instance, see the `EmployeeService` class. This project uses
Spring Data JPA EntityGraph as a dependency to pass an `EntityGraph` object to Spring Boot repository methods.
** https://medium.com/@gdprao/graphql-with-spqr-and-spring-data-jpa-973e50746ad5[GraphQL with SPQR and Spring Data JPA].

- The official way to resolve this problem is by using the batch loading feature with the `@BatchMapping` annotation. Here are some examples:
** https://techdozo.dev/spring-for-graphql-how-to-solve-the-n1-problem/[Spring for GraphQL: How to solve the N+1 Problem?]
** https://blog.stackademic.com/caching-problem-with-graphql-in-spring-boot-8021fe6089d7[N+1 Problem for GraphQL in Spring Boot solved using Batching]

- The https://github.com/nazarimilad/spring-boot-graphql-querydsl-demo[Java demo project] demonstrates efficient fetching of relational data, preventing unnecessary selections,
joins, and the N+1 problem when querying data. For instance, in the
https://github.com/nazarimilad/spring-boot-graphql-querydsl-demo/blob/main/src/main/java/be/nazarimilad/springbootgraphqlquerydsldemo/controller/DeliveryOrderController
.java[DeliveryOrderController.java], the repository method is called with `DataFetchingFieldSelectionSet`: `deliveryOrderRepository.findDeliveryOrder(selectionSet, id)`.
Within `deliveryOrderRepository`, the `queryBuilder.innerJoin` explicitly joins a related entity, thereby avoiding the N+1 query problem.

- In the article https://www.graphql-java.com/blog/deep-dive-data-fetcher-results/[Building Efficient Data Fetchers by Looking Ahead], the idea is to use
`DataFetchingEnvironment.getLocalContext()` and `DataFetchingEnvironment.getSource()` to store prefetched data and pass it to the next data fetcher.

=== With logic on the database layer

In the article https://evilmartians.com/chronicles/soft-deletion-with-postgresql-but-with-logic-on-the-database[Soft deletion with PostgreSQL: but with logic on the database!],
the author presents a pure database solution consiting of:

1. A rule `_soft_deletion` on DELETE table event that sets the column `deleted = true`:
+
[source,sql]
----
CREATE RULE "_delete_orders" AS ON UPDATE TO users
  WHERE NOT old.deleted AND new.deleted
  DO ALSO UPDATE orders SET deleted = true WHERE user_id = old.id;
----

2. The rule may be disabled via settings, thus obtaining the hard deletion:
+
[source,sql]
----
`ALTER SETTINGS SET rules.soft_deletion TO on`;
----
+
[source,sql]
----
CREATE OR REPLACE RULE "_soft_deletion"
  AS ON DELETE TO "orders"
  WHERE current_setting('rules.soft_deletion') = 'on'
  DO INSTEAD UPDATE orders SET deleted = true WHERE id = old.id;
----

== How methods annotated with `@BatchMapping` are registered as batch loaders

When the application starts, the `org.springframework.graphql.execution.DefaultBatchLoaderRegistry` bean is instantiated and `DefaultBatchLoaderRegistry$DefaultRegistrationSpec`'s
method `registerMappedBatchLoader()`` is called in a loop for each method annotated with `@BatchMapping`. The batch data loaders are wrapped into `ReactorMappedBatchLoader`
instances and added to the `mappedLoaders` field of `DefaultBatchLoaderRegistry` bean. This process is done once. During each request, the `registerDataLoaders()` method is called
from the `DefaultExecutionGraphQlService` bean. This method registers all data loaders from the `mappedLoaders` list to the `graphql.DataLoaderRegistry`.

== About pinned java virtual threads

The article https://developers.ascendcorp.com/why-are-my-java-virtual-threads-slower-than-the-platform-threads-74612a1587f3[Why are my Java virtual threads slower than the platform threads?]
introduces the concept of "thread pinning".

"Pinning describes the condition where a virtual thread, which maps to a platform thread, is stuck to the carrier thread.
This means that the virtual thread cannot be unmounted from the carrier thread, preventing others from utilizing the same platform thread."

The causes for a virtual thread to be pinned include:

1. A synchronized block or method
2. A native method
3. A foreign function

These JVM options can be used to detect problematic (pinned) virtual threads: `-Djdk.tracePinnedThreads=short` or `-Djdk.tracePinnedThreads=full`.

One solution recommended by the author is to "replace synchronized with ReentrantLock".
For example, in MySQL Connector/J 9.0.0+, synchronized blocks were replaced with ReentrantLocks, which allows virtual threads to perform optimally with database interactions.
